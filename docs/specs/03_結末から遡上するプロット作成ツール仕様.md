# 03_結末から遡上するプロット作成ツール仕様

## 位置づけ

この文書は、`Structured-storytelling-project` における
`結末から遡上するプロット作成ツール`
の仕様を整理したもの。

本ツールは、一般的な「設定から物語を積み上げる」方式ではなく、
`最後にどのような結末に到達していてほしいか`
を起点に、そこへ至るための人物条件、関係条件、出来事条件、因果条件を逆算していく。

目的は、AI に本文を書かせることではなく、
`人間と AI が共著で、破綻しにくい物語構造とプロットを作ること`
にある。

---

## ゴール

本ツールのゴールは次のとおり。

1. ユーザーが入力した `結末` を起点にする
2. その結末を成立させるための前提条件を、複数候補として遡上する
3. 各段階でユーザーが選択し、補助メモを加えながら物語構造を育てる
4. 途中状態と候補履歴を含む `JSON 正本` を保持する
5. 最終的に、`シーン列レベルのプロット .md` を出力する

ここでいう最終出力は `本文` ではない。
本ツールは `小説生成器` ではなく、`構造共著・プロット設計器` として位置づける。

---

## スコープ

### 本ツールが担うもの
- 結末入力を起点にした逆算フロー
- 複数候補の提示
- ユーザー選択と補助メモの反映
- 物語構造の JSON 正本化
- 人間向けプロット `.md` の出力

### 本ツールが担わないもの
- 小説本文の自動生成
- 完成原稿の文章化
- 断片連結アプリ側での断片展開そのもの

断片連結アプリは、将来的な接続先として想定するが、本仕様の主担当は
`結末 -> 構造 -> プロット`
までである。

---

## 基本思想

このツールの基本思想は次のとおり。

1. `結末` を起点にする
2. 一意の正解を即断せず、複数の逆算ルートを提示する
3. ユーザーが選びながら構造を確定していく
4. 過程も候補履歴として残す
5. 正本は文章ではなく JSON に置く
6. `.md` は正本を人間向けに投影したビューとして扱う

つまり本ツールは、
`結末から存在条件を逆算する物語ハーネス`
として設計する。

---

## 入力

### ユーザーの最初の入力
- `結末`

ここでの `結末` は、少なくとも次のいずれかを含む自然文入力を想定する。

- 誰がどうなっているか
- 何を得たか、何を失ったか
- 誰との関係がどう変わったか
- どういう感情や状態で終わるか

### 各段階での追加入力
- 選択肢の選択
- 補助メモ型の自由記述テキスト

補助メモは主決定入力ではなく、次の候補生成を補正するための材料とする。
たとえば次のような内容を受ける。

- この候補のどこが近いか
- 何が違和感か
- 残したい要素
- 避けたい要素
- 次候補への補正指示

---

## 出力

### 各段階の中間出力
- 逆算の入口候補
- 成立条件候補
- キャラ条件候補
- 関係条件候補
- 必須イベント候補
- 転換点候補

各段で複数候補を提示し、ユーザーが選んだ結果を次段へ引き継ぐ。

### 最終出力
- `シーン列レベルのプロット .md`

出力される `.md` は、本文ではなく、人間が執筆可能な粒度の設計図とする。

各シーンには、少なくとも次を含める。

- シーンの役割
- そのシーンで必要な因果
- 関係変化
- キャラクターが行動する前提
- 破綻してはいけない条件

---

## フロー

### 1. 結末入力
ユーザーが `結末` を入力する。

### 2. 逆算の入口候補提示
ツールは、結末をどういう観点から逆算するかを複数提示する。

想定される入口例:
- 物語構造から入る
- 主人公から入る
- 関係性から入る

### 3. ユーザー選択 + 補助メモ
ユーザーは入口を選択し、必要なら補助メモを入れる。

### 4. 成立条件候補提示
選ばれた入口を前提に、結末成立のための条件候補を複数提示する。

たとえば次のような候補がありうる。
- 必要な欠落
- 必要な欲望
- 必要な誤信念
- 必要な関係配置
- 必要な決定的イベントの種類

### 5. ユーザー選択 + 補助メモ
各候補について、ユーザーが選択・補足する。

### 6. 段階的な遡上
以後も同じ形式で、
`選択肢提示 -> 選択 -> 補助メモ反映`
を繰り返しながら、物語構造を段階的に育てる。

### 7. JSON 正本更新
各ラウンド終了時に、途中状態を JSON 正本へ反映する。

### 8. プロット `.md` 出力
十分に構造が固まった段階で、JSON 正本からシーン列レベルのプロット `.md` を生成する。

---

## JSON 正本の考え方

### 正本にする理由
正本を `.md` ではなく `JSON` に置く理由は次のとおり。

- 物語成立条件を構造化して保持しやすい
- 候補履歴を持てる
- あとから別形式へ変換できる
- 人間と AI が同じ構造を参照できる

### JSON に保持するもの
少なくとも次の層を持つ。

- 終着条件
- 選択した逆算入口
- 成立条件
- キャラの内面条件
- 関係条件
- 必要イベント
- 転換点
- 破綻禁止条件
- シーン列の骨格
- 候補履歴
- ユーザー補助メモ

### 候補履歴の扱い
本ツールは `AI と共著する` ことを重視するため、
`確定事項だけ` を持つのではなく、`比較した候補と採否履歴` も保存する。

候補履歴は薄い要約ではなく、全文保存に近い粒度で保持する。

保持対象の例:
- 提示候補
- 採択候補
- 非採択候補
- ユーザー補助メモ
- 採否に影響した理由

---

## プロット `.md` の考え方

### 役割
`.md` は正本ではなく、JSON 正本を人間向けに投影したビューである。

### 粒度
最終出力は `シーン列レベル` とする。
これは、あらすじより細かく、本文より粗い中間粒度である。

### 含めるべき情報
各シーンで少なくとも次を読めるようにする。

- シーンの目的
- 因果上の役割
- 関係変化
- 行動前提
- 破綻禁止条件

### 含めないもの
- 小説本文そのもの
- 完成原稿に近い文章化
- AI が書いた地の文や会話文の完成形

---

## 断片連結アプリとの関係

本ツールは `設計層`、断片連結アプリは `展開層` と位置づける。

### 本ツール側
- 結末から逆算して構造を作る
- 正本 JSON を保持する
- プロット `.md` を出す

### 断片連結アプリ側
- 出来上がったプロットをもとに、断片や本文へ展開する
- 執筆支援として機能する

ただし本仕様では、断片連結アプリへの接続は将来前提に留める。
現時点の完成条件は `プロット .md` の出力までとする。

---

## AI の役割

AI の役割は次のとおり。

- 複数候補の提示
- 比較観点の提示
- 因果破綻の検知補助
- 前提条件の遡上支援
- シーン列レベルの構造整理

AI が担わないものは次のとおり。

- 本文の最終執筆
- 作者の代替
- 一意の正解の押し付け

本ツールは、AI が一方的に生成するのではなく、
`比較提案能力を活かして人間の構想を支える` 設計とする。

---

## 成功条件

本仕様が成立したと言える条件は次のとおり。

1. 結末入力から逆算フローを始められる
2. 入口候補や成立条件候補が複数提示される
3. ユーザーの選択と補助メモが次候補生成に反映される
4. 途中状態と候補履歴を含む JSON 正本が保持される
5. JSON 正本からシーン列レベルのプロット `.md` を生成できる
6. `.md` は本文ではなく、執筆用の設計図として読める
7. 因果や行動前提の破綻を抑える構造が前面化されている

---

## 一言でまとめると

このツールは、
`結末を起点に、AI と人間が複数候補を比較しながら物語成立条件を遡上し、JSON 正本として保持し、最終的にシーン列レベルのプロット .md を出力するための共著型プロット設計ツール`
である。
