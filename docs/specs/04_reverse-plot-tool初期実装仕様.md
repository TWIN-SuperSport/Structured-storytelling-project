## 1. 概要

`Structured-storytelling-project` は、物語の結末や終着条件から逆算して、
成立条件・人物条件・関係条件・出来事条件を構造化し、
JSON 正本と人間向けプロットを育てる親プロジェクト。

その最初の deployable child service が `reverse-plot-tool`。
現時点では、結末の自由文から物語構造 JSON と短いプロット骨格を返す最小版として実装している。
現行運用では、単発で重いプロット生成に寄せるため、`Swallow-8B` 系の専用 relay を使う。

**設計方針**
- 初期版は `reverse-plot-tool` を単機能の検証サービスとして切り出す
- 正本は文章ではなく JSON に置く
- UI は最小でよい。まずは入力、生成、JSON 確認を優先する
- 将来的には「複数候補提示」「選択」「補助メモ反映」の共著フローへ拡張する

## 2. 画面仕様

### 基本構成

| エリア | 役割 |
|---|---|
| 左側（入力エリア） | 終着条件、ジャンルヒント、主人公ヒントを入力 |
| 右側（結果エリア） | 生成された JSON をそのまま表示 |

### 入力エリア

**主要入力**
- `終着条件 / 結末`
- `ジャンルヒント`
- `主人公ヒント`

**ボタン**

| ボタン | 動作 |
|---|---|
| `物語構造を遡上する` | 終着条件から物語構造 JSON を生成 |

### 結果表示

- 現在は JSON をそのまま `pre` 表示する
- 最終的にはこの JSON をもとに `.md` のプロットビューへ投影する

### 初期版で不要なもの

複数候補選択 UI、候補履歴 UI、ログイン、保存一覧、本文生成 UI

## 3. コンポーネント構成

| コンポーネント | 役割 | 持たせない責務 |
|---|---|---|
| UI | 入力受付・送信・JSON 表示・エラー表示 | LLM 直接接続・永続保存 |

## 4. API 仕様

### 一覧

| API | メソッド | パス | 用途 |
|---|---|---|---|
| 逆算プロット生成 | `POST` | `/api/story/reverse-plot` | 終着条件から物語構造 JSON を生成 |
| ヘルスチェック | `GET` | `/api/health` | ミドルウェア稼働確認 |

### POST /api/story/reverse-plot

**リクエスト**

| フィールド | 型 | 必須 | 説明 |
|---|---|---|---|
| `ending_text` | string | 必須 | 終着条件 / 結末の自由文 |
| `protagonist_hint` | string | 任意 | 主人公ヒント |
| `genre_hint` | string | 任意 | ジャンルヒント |
| `output_format_version` | string | 任意 | 出力 schema の版 |

```json
{
  "ending_text": "現代日本から異世界に転生した高校生の男子が、異世界で魔法使いとして成長し、魔王を討伐したあと、日本に再転生して終わる。",
  "genre_hint": "異世界転生ファンタジー",
  "protagonist_hint": "男子高校生"
}
```

**成功レスポンス**

| フィールド | 必須 | 説明 |
|---|---|---|
| `status` | 必須 | `"success"` |
| `story.title` | 必須 | 仮タイトル |
| `story.ending_summary` | 必須 | 結末要約 |
| `story.core_theme` | 必須 | 中心テーマ |
| `story.protagonist_final_state` | 必須 | 結末時の主人公状態 |
| `story.structural_conditions` | 必須 | 初期欠落、欲望、恐れ、誤信念、初期配置 |
| `story.relationship_changes` | 必須 | 必要な関係変化 |
| `story.required_turning_points` | 必須 | 必要な転換点 |
| `story.failure_conditions` | 必須 | 破綻禁止条件 |
| `story.plot` | 必須 | `prologue`, `ki`, `sho`, `ten`, `ketsu`, `epilogue` |

      "必要な関係変化1",

    ],
    "required_turning_points": [
      "必要な転換点1",
      "必要な転換点2",
      "必要な転換点3"
    ],
    "failure_conditions": [
      "破綻してはいけない条件1",
      "破綻してはいけない条件2"
    ],
    "plot": {
      "prologue": "2〜3文のあらすじ",
      "ki": "2〜3文のあらすじ",

**エラーレスポンス**

| HTTP | status | 原因 |
|---|---|---|
| 400 | `invalid_request` | 入力空・文字数超過 |
| 422 | `invalid_llm_output` | LLM 出力が schema 不適合 |
| 502 | `llm_upstream_error` | LLM 呼び出し失敗 |
| 504 | `llm_timeout` | LLM 応答タイムアウト |

## 5. 状態遷移

### 状態一覧

| 状態 | 意味 | UI の見え方 |
|---|---|---|
| `idle` | 初期状態 | 入力済みサンプルと空の結果欄 |
| `editing` | 入力中 | テキスト編集可能 |
| `submitting` | 生成要求送信中 | `考え中...` 表示 |
| `success` | 生成成功 | JSON 表示 |
| `error` | 生成失敗 | エラー文字列表示 |

### 主要な状態遷移

| 現在状態 | イベント | 次状態 |
|---|---|---|
| `idle` / `editing` | ボタン押下 | `submitting` |
| `submitting` | API 正常応答 | `success` |
| `submitting` | 通信失敗 / schema 不適合 / timeout | `error` |
| `success` | 入力変更 | `editing` |
| `error` | 入力修正 / 再送 | `editing` / `submitting` |

## 6. 本来フローと初期実装の差分

### 本来ほしいフロー

上流仕様では、本ツールは次のフローを想定している。

1. 結末入力
2. 逆算入口候補提示
3. ユーザー選択
4. 成立条件候補提示
5. 補助メモ反映
6. 段階的な遡上
7. JSON 正本更新
8. シーン列レベルの `.md` 出力

### 現在の初期実装

現時点では、
- 終着条件を1回受ける
- LLM に一括逆算させる
- JSON を返す

までを最小実装としている。

つまり、
**本来の対話的な複数候補フローはまだ未実装**
であり、今は最終形へ向けた足場の段階。
